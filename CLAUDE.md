# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What This Repo Is

**magic-seed is a markdown-only project. There is no code, no build system, no test suite, no CLI.**

It is a *workflow generator* whose runtime is the AI itself. All artifacts are prose markdown files that an AI assistant reads and acts on. magic-seed is platform-agnostic: the workflow logic lives in `instructions.md` at the repo root, and per-platform wrappers in `platforms/{tool}/` adapt that logic to each AI's discovery and trigger conventions (Claude Code skill YAML, Cursor `.mdc` rules, GitHub Copilot always-on instructions, OpenCode skills, Kimi Code `AGENTS.md`).

When a user installs magic-seed and asks the AI to initialize it, the AI:

1. Detects the project type by scanning `profiles/*/README.md` detection hints
2. Loads `universal/rules.md` (always) + `profiles/{detected}/{README,discovery,rules}.md`
3. Executes the discovery script in `discovery.md` to fill a slot catalog
4. Renders skeletons from `profiles/{detected}/skeletons/*.md` into `.ai-workflow/wizards/*.md` in the user's project
5. Initializes the three-tier knowledge base in `docs/`

**The entry point is `instructions.md` at the repo root.** Read it first to understand the orchestration contract. Per-platform wrappers in `platforms/{tool}/` are intentionally thin and reference it.

## Architecture

Four layers, strictly ordered (more universal → more specific):

| Layer | Path | Purpose | Modify when |
|-------|------|---------|-------------|
| Universal | `instructions.md` + `universal/*.md` | Platform-neutral workflow logic + non-negotiable rules and structure | Adding a rule or workflow change that applies to every tech stack and every platform |
| Profile | `profiles/{type}/` | Per-tech-stack detection, discovery, conventions, wizard skeletons | Adding/improving support for a tech stack |
| Platform | `platforms/{tool}/` | Per-AI-tool packaging: YAML format, trigger semantics, install instructions | Adding/improving support for an AI assistant |
| User project | `.ai-workflow/` (in user repo) + `docs/features/{name}/` | Generated wizards + per-feature knowledge base | Generated by the system, not maintained here |

### Universal layer (`instructions.md` + `universal/`)

- `instructions.md` — Repo-root entry point. The platform-neutral version of the workflow that every wrapper references. Drop this and the system has no orchestration.
- `universal/rules.md` — 15 numbered rules with severity (`error` / `warning` / `info`). Rules 1, 2, 4, 6, 8, 9 are blocking. **When adding a rule, follow the existing numbered structure** and assign a severity in the "Enforcement" section at the bottom.
- `universal/workflow-structure.md` — Defines the 6 wizard types and their phase architecture (e.g., design has UNDERSTAND → DESIGN → SPECIFY → REVIEW → PLAN). Every phase ends with an `[A]ccept / [F]eedback / [R]eject` gate.
- `universal/diagram-standards.md` — PlantUML is the default; Mermaid and D2 are alternatives.
- `universal/knowledge-base-spec.md` — Three-tier KB: feature-level (`docs/features/{name}/`), project-level (`docs/project/`), team-level (`docs/team/`).
- `universal/slot-catalog-template.md`, `universal/wizard-authoring-guide.md` — Templates and authoring conventions for new profiles/wizards.

### Profile layer (`profiles/{type}/`)

Each profile MUST have these four files (the system reads them by name):

- `README.md` — Detection hints (file/directory indicators + confidence scoring) and capability summary. The detection block is parsed by the AI during initialization.
- `discovery.md` — Step-by-step instructions for the AI to explore a codebase of this type and fill named **slots** (`project-name`, `source-directory`, etc.).
- `rules.md` — Tech-stack-specific conventions layered on top of `universal/rules.md`.
- `skeletons/*.md` — Wizard templates with `{slot-name}` placeholders. Standard set is six files: `design-wizard.md`, `implement-wizard.md`, `pr-wizard.md`, `test-wizard.md`, `deploy-wizard.md`, `docs-wizard.md`.

Existing profiles: `generic` (low-confidence fallback), `godot-game`, `unity-game`, `web-frontend`, `backend-api`, `mobile-app`.

**Note:** Each profile also contains an empty `{skeletons}/` directory (literal name with braces) alongside the real `skeletons/`. These appear to be template-rendering leftovers; if you create a new profile by copying an existing one, leave them or delete them — they're not referenced by `instructions.md` or any wrapper.

### Platform layer (`platforms/{tool}/`)

Each platform wrapper packages the universal workflow for a specific AI tool. Wrappers are intentionally thin (~20–80 lines) and reference `instructions.md` for the actual logic — they encode only the tool's discovery / trigger semantics and its install path.

| Tool | Wrapper file | Install target | Trigger model |
|------|-------------|----------------|---------------|
| Claude Code | `platforms/claude/SKILL.md` | `.ai-workflow/SKILL.md` | YAML `description` matches user request |
| Cursor | `platforms/cursor/magic-seed.mdc` | `.cursor/rules/magic-seed.mdc` | "Apply Intelligently" — agent reads `description` |
| GitHub Copilot | `platforms/github-copilot/copilot-instructions.md` | `.github/copilot-instructions.md` | Always-on |
| OpenCode | `platforms/opencode/SKILL.md` | `.opencode/skills/magic-seed/SKILL.md` | Skill loaded on demand by `description` |
| Kimi Code CLI | `platforms/kimi-code/AGENTS.md` | `AGENTS.md` (project root) | Manual reference today (`@AGENTS.md`); auto-load pending upstream |

Each wrapper directory also has a `README.md` documenting the install path, symlink-vs-copy options, append-don't-overwrite paths for projects with conflicting files, and any platform-specific caveats.

**Hard rule for wrappers:** they must not duplicate workflow logic. If a wrapper grows beyond install paths, trigger config, and a pointer at `instructions.md`, the additions belong in `instructions.md` instead.

## Working in This Repo

There is nothing to build, lint, or test. Common tasks:

- **Add a new profile** — Copy `profiles/generic/` to `profiles/{new-type}/`, edit the four files (`README.md` detection hints, `discovery.md` slot scripts, `rules.md` conventions, `skeletons/*.md` templates). See `docs/how-to-create-profile.md`.
- **Add a universal rule** — Append to `universal/rules.md` with a new rule number and severity. Update the "Enforcement" / severity block at the bottom.
- **Add a new wizard type** — Add a `skeletons/{wizard-name}.md` to each profile that should support it, following the phase + gate pattern in `universal/workflow-structure.md`. If the new wizard should be globally discoverable, also document its intent in `instructions.md`.
- **Add a new platform wrapper** — Create `platforms/{new-tool}/` with the tool's native instruction file (whatever YAML/markdown convention it uses) and a `README.md` install guide. Wrapper body should point at `instructions.md` and list the magic-seed intents that should trigger workflow behavior — nothing more. Update the platform table in this file and the install table in `README.md`.
- **Change diagram conventions** — Edit `universal/diagram-standards.md`. PlantUML is the default; do not switch defaults without strong reason.

### Validation (manual, since there's no test runner)

When editing wizards, skeletons, rules, or wrappers, manually check:

1. **No leaked placeholders** — Per Rule 12, no `{[a-z][a-z0-9_-]*}` patterns should remain in non-template files. Template files (skeletons, `*-TEMPLATE.md`) are exempt.
2. **Slot names match across profile** — Slot names referenced in `skeletons/*.md` must be defined/discovered in the profile's `discovery.md`.
3. **Phase gates intact** — Every wizard phase ends with an explicit `[A]/[F]/[R]` gate. Do not remove these.
4. **File reading order in `instructions.md`** — If you add new universal files, update the "File Reading Order" section in `instructions.md`.
5. **No slash-command syntax outside `platforms/claude/`** — magic-seed is platform-neutral. Only the Claude wrapper's command-alias table (in `platforms/claude/SKILL.md`) may use slash invocations of the magic-seed prefix or per-wizard names. Everywhere else, refer to wizards by name in prose. Verify by grepping for the relevant slash patterns and excluding `platforms/`; the GitHub clone URL ending in `.git` is a known harmless false positive.
6. **Wrappers stay thin** — If a `platforms/{tool}/*` file grows beyond the wrapper's expected ~20–80 lines, the additions probably belong in `instructions.md`. Wrappers may include a short "Init must persist plumbing" reminder pointing at `instructions.md` Step 7 — that's the one allowed exception, since the bug it prevents (generate-and-forget init) is wrapper-discoverable but not wrapper-fixable.
7. **Init persistence** — When changing the init flow, preserve the contract that init produces *all four* artifacts: platform wrapper, universal/profile reachable, rendered wizards, knowledge base scaffold. If you find a wrapper README or SKILL/AGENTS file that lets the agent stop after rendering wizards, it's a bug — the wrapper must point at `instructions.md` Step 7 + Step 8 explicitly.
8. **Tool ≠ model** — `platforms/{tool}/` directories are keyed on the AI tool (Claude Code, OpenCode, Cursor, Kimi-Code CLI, GitHub Copilot), not the LLM provider. When extending Step 0 in `instructions.md` or adding detection signals, use tool-level signals (`.opencode/`, `.cursor/`, etc.), never model names.
9. **Profile dirs are exactly four entries** — every `profiles/{type}/` contains exactly `README.md`, `discovery.md`, `rules.md`, and `skeletons/`. No literal-name placeholder dirs (e.g. `{skeletons}/`), no extras. Step 7.2 now propagates profile dirs into user projects, so anything stray here ends up in their tree. Verify with `find profiles -type d -name '{*}'` — must return empty.
10. **Skill wrappers alone are inert on pull-not-push platforms** — when adding/editing a wrapper, remember that OpenCode (and likely future tools) expose skills as tools the agent decides to invoke, not as auto-loaded context. The skill's YAML `description` must be imperative ("MUST be loaded when..."), not descriptive ("Universal workflow generator..."), AND Step 7.6 must write a magic-seed directive to the project's default-context file (`AGENTS.md` for OpenCode/Cursor, `CLAUDE.md` for Claude Code). Without the directive, the agent has no project-level pressure to defer to wizards on natural-language requests and silently improvises feature work — making the install functionally inert.

## Hard Rules to Preserve When Editing

These are load-bearing for the whole system; don't soften them without discussing:

- **No auto-approval** (Rule 6) — The AI never decides a phase is complete. Always `[A]/[F]/[R]`.
- **DESIGN.md is immutable after sign-off** (Rule 9) — Deviations go in TDD.md, never edit a signed-off DESIGN.md.
- **Diagrams are mandatory** (Rule 4) — Every feature gets at least a class + package diagram.
- **Living documents, no archives** (Rule 2) — Update in place. Never create `docs-v2/` or dated copies.
- **Universal-first reading order** — Wizards must load `universal/rules.md` before profile rules before project overrides.
- **Wrappers don't own logic** — All workflow logic lives in `instructions.md` and the universal/profile files. Wrappers translate platform triggers into intents, nothing more.
