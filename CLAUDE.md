# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What This Repo Is

**magic-seed is a markdown-only project. There is no code, no build system, no test suite, no CLI.**

It is a *workflow generator* whose runtime is the AI itself. All artifacts are prose markdown files that an AI assistant (Claude, Cursor, etc.) reads and acts on. When a user runs `/magic-seed init` in their own project (after cloning this repo to `.ai-workflow/`), the AI:

1. Detects the project type by scanning `profiles/*/README.md` detection hints
2. Loads `universal/rules.md` (always) + `profiles/{detected}/{README,discovery,rules}.md`
3. Executes the discovery script in `discovery.md` to fill a slot catalog
4. Renders skeletons from `profiles/{detected}/skeletons/*.md` into `.ai-workflow/wizards/*.md` in the user's project
5. Initializes the three-tier knowledge base in `docs/`

**The platform-neutral entry point is `instructions.md` at the repo root.** Per-platform wrappers live in `platforms/{tool}/` and reference it. Read `instructions.md` first to understand the orchestration contract.

## Architecture

Three layers, strictly ordered (more universal → more specific):

| Layer | Path | Purpose | Modify when |
|-------|------|---------|-------------|
| Universal | `universal/*.md` | Non-negotiable rules + workflow structure shared by all profiles | Adding a rule that applies to every tech stack |
| Profile | `profiles/{type}/` | Per-tech-stack detection, discovery, conventions, wizard skeletons | Adding/improving support for a tech stack |
| User project | `.ai-workflow/` (in user repo) + `docs/features/{name}/` | Generated wizards + per-feature knowledge base | Generated by the system, not maintained here |

### Universal layer (`universal/`)

- `rules.md` — 15 numbered rules with severity (`error` / `warning` / `info`). Rules 1, 2, 4, 6, 8, 9 are blocking. **When adding a rule, follow the existing numbered structure** and assign a severity in the "Enforcement" section at the bottom.
- `workflow-structure.md` — Defines the 6 wizard types and their phase architecture (e.g., design has UNDERSTAND → DESIGN → SPECIFY → REVIEW → PLAN). Every phase ends with an `[A]ccept / [F]eedback / [R]eject` gate.
- `diagram-standards.md` — PlantUML is the default; Mermaid and D2 are alternatives.
- `knowledge-base-spec.md` — Three-tier KB: feature-level (`docs/features/{name}/`), project-level (`docs/project/`), team-level (`docs/team/`).
- `slot-catalog-template.md`, `wizard-authoring-guide.md` — Templates and authoring conventions for new profiles/wizards.

### Profile layer (`profiles/{type}/`)

Each profile MUST have these four files (the system reads them by name):

- `README.md` — Detection hints (file/directory indicators + confidence scoring) and capability summary. The detection block is parsed by the AI during `/magic-seed init`.
- `discovery.md` — Step-by-step instructions for the AI to explore a codebase of this type and fill named **slots** (`project-name`, `source-directory`, etc.).
- `rules.md` — Tech-stack-specific conventions layered on top of `universal/rules.md`.
- `skeletons/*.md` — Wizard templates with `{slot-name}` placeholders. Standard set is six files: `design-wizard.md`, `implement-wizard.md`, `pr-wizard.md`, `test-wizard.md`, `deploy-wizard.md`, `docs-wizard.md`.

Existing profiles: `generic` (low-confidence fallback), `godot-game`, `unity-game`, `web-frontend`, `backend-api`, `mobile-app`.

**Note:** Each profile also contains an empty `{skeletons}/` directory (literal name with braces) alongside the real `skeletons/`. These appear to be template-rendering leftovers; if you create a new profile by copying an existing one, leave them or delete them — they're not referenced by `SKILL.md`.

## Working in This Repo

There is nothing to build, lint, or test. Common tasks:

- **Add a new profile** — Copy `profiles/generic/` to `profiles/{new-type}/`, edit the four files (`README.md` detection hints, `discovery.md` slot scripts, `rules.md` conventions, `skeletons/*.md` templates). See `docs/how-to-create-profile.md`.
- **Add a universal rule** — Append to `universal/rules.md` with a new rule number and severity. Update the "Enforcement" / severity block at the bottom.
- **Add a new wizard type** — Add a `skeletons/{wizard-name}.md` to each profile that should support it, following the phase + gate pattern in `universal/workflow-structure.md`.
- **Change diagram conventions** — Edit `universal/diagram-standards.md`. PlantUML is the default; do not switch defaults without strong reason.

### Validation (manual, since there's no test runner)

When editing wizards, skeletons, or rules, manually check:

1. **No leaked placeholders** — Per Rule 12, no `{[a-z][a-z0-9_-]*}` patterns should remain in non-template files. Template files (skeletons, `*-TEMPLATE.md`) are exempt.
2. **Slot names match across profile** — Slot names referenced in `skeletons/*.md` must be defined/discovered in the profile's `discovery.md`.
3. **Phase gates intact** — Every wizard phase ends with an explicit `[A]/[F]/[R]` gate. Do not remove these.
4. **File reading order in `instructions.md`** — If you add new universal files, update the "File Reading Order" section in `instructions.md` at the repo root.

## Hard Rules to Preserve When Editing

These are load-bearing for the whole system; don't soften them without discussing:

- **No auto-approval** (Rule 6) — The AI never decides a phase is complete. Always `[A]/[F]/[R]`.
- **DESIGN.md is immutable after sign-off** (Rule 9) — Deviations go in TDD.md, never edit a signed-off DESIGN.md.
- **Diagrams are mandatory** (Rule 4) — Every feature gets at least a class + package diagram.
- **Living documents, no archives** (Rule 2) — Update in place. Never create `docs-v2/` or dated copies.
- **Universal-first reading order** — Wizards must load `universal/rules.md` before profile rules before project overrides.
